## 2세션: period=32 레이턴시 최적화

### sec_path 재측정 (period=32)

- 파라미터: `--period=32 --buffer-mult=4 --start-fill=1 --xrun-fill=2`, DSP=IIR(mode 1)
- 결과: `S_hat(final): peak=0.0114 @ 135 (2.81ms)` — 이전 218샘플(4.54ms) 대비 **1.73ms 단축**
- `runtime/snore_sec_path.bin` 갱신 완료

### NB ANC 루프 write-before-read 수정

- **문제**: period=32에서 NB 모드 실행 시 capture xrun이 연속 발생
- **원인**: HiFiBerry linked I2S 스트림은 START 이후 WRITEI → READI 순서 필수. 기존 코드가 read-first 순서였음
- **수정**: `anc.c` NB while 루프를 write-first 순서로 변경 (measure 모드와 동일한 구조로)
- **결과**: 시작 시 xr=1(정상), 이후 안정

### anc-snore.service 파라미터 확정

```
ExecStart=.../snore_anc nb --period=32 --buffer-mult=4 --start-fill=1 --xrun-fill=2 --n-harm=4 --nb-mu=0.0005 --nb-leak=0.9999
```

### 추가 레이턴시 단축 시도 (모두 실패/롤백)

| 시도 | 결과 | 이유 |
|------|------|------|
| `--xrun-fill=1` | 연속 playback xrun | 하드웨어 최솟값이 2 period (64샘플) |
| PCM5122 DSP mode 3 | 스피커 무음 | DSP 계수 없이 "Fixed process flow" → 출력 없음 |

- 두 시도 모두 즉시 롤백. `xrun-fill=2` / mode 1 이 현재 하드웨어에서의 한계

### 스피커 무음 문제 (미해결 → 재부팅 필요)

- mode 3 시도 후 PCM5122 내부 레지스터 오염 추정
- 소프트웨어 진단 결과: mixer 설정 이상 없음, 드라이버 리로드 시도했으나 DT overlay 참조로 완전 재초기화 불가
- **해결책**: 재부팅으로 PCM5122 cold reset 필요. 재부팅 후 anc-snore 서비스 재시작하면 정상 복구 예상

## 3세션: /home/admin/anc.c SW latency 최적화 + Acoustic path 분해

> 대상 코드: `/home/admin/anc.c` (FxLMS broadband ANC, 서비스: `anc-rt.service`)
> 이 코드는 snore_anc(NB)와 별개의 독립 바이너리다.

### 목표 정의 (세션 초반 확정)

| 구분 | 정의 | 이번 세션 대상 |
|------|------|---------------|
| SW/operational latency | readi→compute→writei + ALSA buffering | 대상 |
| Acoustic secondary-path latency | sec_path.bin peak 기준 (~5.69ms) | 별도 지표, 이번 대상 아님 |

### 코드 변경 (anc.c, 2건)

**1. playback start_threshold 분리**

- `set_sw_params()`에 `start_thr` 파라미터 추가
- capture: `start_threshold = period` (기존 유지)
- playback: `start_threshold = buf` (buffer 전체 크기)
- 효과: prefill 중 playback auto-start 방지 → startup xrun 제거

```c
// 변경 전
set_sw_params(a->cap,  cap_period,  cap_buf);
set_sw_params(a->play, play_period, play_buf);

// 변경 후
set_sw_params(a->cap,  cap_period, cap_buf,  cap_period);   // capture: auto-start OK
set_sw_params(a->play, play_period, play_buf, play_buf);    // playback: manual start only
```

**2. SCHED_FIFO priority 49 추가**

- `rt_prepare_memory()`에 `sched_setscheduler(0, SCHED_FIFO, &sp)` 추가
- `#include <sched.h>` 추가
- 효과: `sudo ./anc` 직접 실행 시도 RT 스케줄링 보호
- 서비스 환경(anc-rt.service)에서는 `CPUSchedulingPriority=70`이 상위라서 코드 설정은 무시됨

### 서비스 변경 (anc-rt.service)

```
# 변경 전
ExecStart=... --start-fill=3 --xrun-fill=3 ...

# 변경 후
ExecStartPre=/usr/bin/amixer -c sndrpihifiberry cset "name=DSP Program" 1   ← 추가
ExecStart=... --start-fill=2 --xrun-fill=3 ...   ← start-fill=3→2
```

### SW latency 안정화 테스트

| 테스트 | 시간 | 모드 | 조건 | xr | 판정 |
|--------|------|------|------|-----|------|
| 직접 실행 | 15s | passthrough | period=64, sf=2, xf=3 | 0 | PASS |
| 직접 실행 | 64s | run | period=64, sf=2, xf=3 | 0 | PASS |
| 서비스 | 895s | run | CPU3, FIFO70, sf=2, xf=3 | 0 | PASS |

steady-state 수치 (period=64, run 모드):

| 지표 | avg | max |
|------|-----|-----|
| read | 1.020ms | 1.095ms |
| compute | 0.307ms | 0.351ms |
| write | 0.007ms | 0.014ms |
| total | 1.333ms | 1.466ms |

**SW operational latency < 2ms 목표 달성.**

### DAC DSP Filter 비교 실측

```bash
amixer -c sndrpihifiberry cset 'name=DSP Program' N
```

| N | 필터명 | sec_path peak (period=64) | 차이 |
|---|--------|---------------------------|------|
| 0 | FIR interpolation (기본) | 211 (4.40ms) | baseline |
| **1** | **Low latency IIR** | **194 (4.04ms)** | **-17샘플 (-0.35ms)** |
| 4 | Ringing-less low latency FIR | 193 (4.02ms) | -18샘플 |

→ **Low latency IIR (mode 1) 채택.** 서비스 ExecStartPre에 영구 반영.

### Secondary Path 분해 (정량 실측)

period 별 sec_path peak 변화를 이용해 구성 요소 분리:

| 구성 요소 | 추정 샘플 수 | 추정 시간 |
|-----------|-------------|-----------|
| ALSA period alignment (64샘플) | ~64 | ~1.33ms |
| DAC digital filter (LL IIR vs FIR 차이) | ~17 | ~0.35ms |
| DAC analog + amp + air + mic + ADC | ~111 | ~2.31ms |
| **합계 (period=64, LL IIR)** | **192** | **4.00ms** |

- 순수 공기 전달 추정: ~1.71ms → speaker-error mic 거리 약 **59cm**
- sec_path.bin 갱신 완료 (peak=192 @ 4.00ms)

### xrun-fill=2 단독 시험 (period=64)

```bash
sudo ./anc passthrough --period=64 --xrun-fill=2   # 29s
```

- xr=0 PASS, startup xrun 없음
- total avg 1.333ms, max 1.408ms — period=64 안정 범위 내
- 결론: `xrun-fill=2`도 period=64에서는 안전. 현재 서비스는 xf=3으로 운영 중 (보수적 유지).

### period=32 실험

```bash
sudo ./anc passthrough --period=32 --start-fill=3 --xrun-fill=3   # 29s xr=0 PASS
sudo ./anc run        --period=32 --start-fill=3 --xrun-fill=3   # 29s xr=0 PASS
```

| 지표 | period=64 | period=32 | 차이 |
|------|-----------|-----------|------|
| sec_path peak | 194 (4.04ms) | 129 (2.69ms) | **-1.35ms** |
| SW total avg | 1.333ms | 0.667ms | **-0.667ms** |
| compute avg | 0.307ms | 0.153ms | 반감 (샘플 수 비례) |
| start-fill 요구 | 2 | **3** | sf=2로는 startup xrun |

→ period=32는 soak test 미완. 서비스 배포 보류.

### Causality(인과성) 분석 결과

```
Ambient xcorr: lag=0 samples — feedforward 조건 미충족
```

| 항목 | 값 |
|------|-----|
| SW processing (1period + compute) | 1.333 + 0.307 = 1.64ms |
| secondary path (speaker→error mic) | 4.00ms |
| **필요한 primary path delay** | **>5.64ms** |
| 실제 ref-error xcorr lag | **0 samples** |
| **causal margin** | **없음 (부족)** |

→ ref mic가 error mic보다 먼저 noise를 감지하지 못함. 이것이 ANC reduction이 +1~5dB에 머무는 근본 원인.
